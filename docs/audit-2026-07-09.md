# Isaac-AI Deep Audit — 2026-07-09

Post-mortem of the 2026-07-06 → 2026-07-08 training runs and the code paths that
produced them. Read this before proposing further changes.

**Scope**: env / protocol / mod bridge, reward shaping, world model + KL,
actor/critic + reinforce, config XS. Empirical evidence pulled from
`tb_dreamer_stage1_20260708-233319.json` (157 k steps, 15.7 h, 339 episodes).

**Bottom line**: the training pipeline has ~4 structural issues stacked on top
of each other. Each surface-level patch has exposed the next one. HP-based
death detection (commit `8d72114`) removes the largest immediate blocker, but
three others remain and any one of them can produce the "actor entropy stuck
at 3.78" symptom on its own.

---

## §2. Design flaws by area

### §2.1 Env / protocol / mod bridge

#### 2.1.1 (P0) Crash penalty conflates two very different events

`env.py:275-311` — when `send_frame` or `recv_frame` throws `ConnectionError`
or `OSError`, we apply `crash_penalty = -1.0` and terminate. **This exact same
path fires for both**:

1. Isaac process actually died (`fleet.respawn` needed)
2. Mod deliberately cycled its socket during a clean in-process restart on
   player death (mod is fully alive, will reconnect within ~500 ms)

The code already knows which case it is — `_try_accept_after_close(wait_s=3.0)`
at line 340 tells us whether the mod reconnected on its own (case 2) or not
(case 1). But this classification is **discarded**: we apply `-1.0` in both
cases. In the 2026-07-08 run:

- `ep_end_mod_socket_error_frac = 0.90` mean (289 of 322 episodes end here)
- `reward/crash_penalty = −0.90` mean per episode

That's a `−0.9` baseline drag on every episode's return. Even after HP-based
death detection (`8d72114`) fires and terminates cleanly on the shaper side,
the *next* env step still hits this path because the mod is cycling its socket
in the background — meaning the terminal episode gets its proper `r_death =
−3` PLUS an extra `-1.0` from the next-step crash detection.

**Fix**: split the two cases explicitly. If `_try_accept_after_close` returned
successfully (Isaac was alive), do NOT apply `crash_penalty`. Instead, treat
the socket cycle as an expected empty transition and either drop the step or
attribute it to the last shaper-terminated episode. Only apply `-1.0` when
`_try_accept_after_close` returned None AND the respawn callback had to fire.

**Rollback**: revert `env.py` change; already-in-tree `ep_end_reason` field
distinguishes the two cases in future runs, so we can audit reward totals
either way.

#### 2.1.2 (P1) The mod's terminal-obs delivery is architecturally fragile

Even after `f340617` (HP-transition trigger + immediate render-side fallback),
the 2026-07-08 run showed `shaper_terminated_frac` peak of 1.0 for the first
episode then collapse to 0. Something about the state between episodes breaks
the delivery path. Without the Isaac `log.txt`, we can't identify the exact
failure mode.

The right long-term fix is what commit `8d72114` did — decouple training from
the mod's terminal signal entirely by inferring death from the regular obs's
HP fields. Once that's verified working, the mod-side death handler becomes
a nice-to-have optimization (one tick earlier signal), not a correctness
requirement.

**However** — `8d72114` is not yet verified. If it has a bug, we're still
stuck. Highest immediate value: run a 15 min smoke test with the current tree
and confirm `shaper_terminated_frac > 0.5`.

#### 2.1.3 (P2) `_RECV_TIMEOUT_S = 1.0` retries forever

`protocol.py:19` — `recv_frame` retries on socket timeout indefinitely, only
raising when the peer actually closes the connection. This means a hung mod
(not crashed, just wedged) produces an *infinite* Python-side block. The
training loop can hang for hours before anyone notices.

**Fix**: track total elapsed time in `recv_exact` and raise `TimeoutError`
after some hard limit (say 30 s). Trainer treats this as a crash and respawns.

**Rollback**: trivial revert.

### §2.2 Reward shaping (`reward.py`)

#### 2.2.1 (P0) `clear_idle_extra` dominates the negative reward budget

Data from 2026-07-08 run:

| Component | Mean per episode | Share of negative side |
|-----------|-----------------:|-----------------------:|
| `clear_idle_extra` | −2.83 | **65%** |
| `crash_penalty` | −0.90 | 21% |
| `hp_delta_red` | −0.48 | 11% |
| `backtrack` | −0.05 | 1% |
| `death` | −0.04 | 1% |
| positive total | +3.93 | — |
| **net** | **−0.42** | — |

`r_clear_room_idle_extra = -0.03` fires per tick where room is clear and
`speed < 0.5`. With ~94 idle ticks per episode, this is −2.83.

**Why this is a design flaw, not just a magnitude issue**: `clear_idle_extra`
is a shaping term designed to encourage forward progress. But it fires on
**every idle tick after room clear** — a random policy will hit this in
almost every episode because a uniformly-random agent stays roughly put
between action commands. So the "penalty" is effectively a constant baseline
drag that carries no policy-dependent gradient signal. **It looks like
shaping but acts like a bias term.**

The consequence is that even a *good* policy (find enemies, kill them,
navigate to door) gets the same −2.83 from this term because it also idles
briefly after killing everything before locating the next door. The bad
policy (do nothing) gets exactly the same value. Zero learning signal
extractable from this component.

**Fix options** (in order of preference):

1. Gate `clear_idle_extra` on `speed < idle_speed_threshold AND
   ticks_since_room_clear > K`. Give the agent 30-60 ticks to "look around"
   after a clear before penalizing idle.
2. Change from tick-based to event-based: apply once per second of idle,
   not once per tick.
3. Scale it way down (0.001 instead of 0.03) so it's a nudge, not the
   dominant term.

I'd start with option 1 (`K = 30`) plus option 3 (`-0.005`).

#### 2.2.2 (P1) Reward heads: episode magnitudes and dynamic range

DreamerV3's value head is a `DiscDist` (twohot symlog) with 255 atoms on
range `[value_v_min=-20, value_v_max=20]`. Per-episode rewards on this run:

- `ep_reward`: mean −0.51, min −4.2, max 2.77
- `ep_reward_best`: 31.2

Undiscounted return magnitudes are within ±31, well inside the `[-20, 20]`
symlog range (symlog compresses beyond that). Fine.

**But**: the range is set assuming stable per-episode return. Right now the
sign of return is dominated by whether crash_penalty fired (−0.9 offset).
Post-fix #2.1.1, the range should be *rechecked* — if fixed, the mean return
should shift up by ~+0.9 and the value head is still fine.

#### 2.2.3 (P1) `r_pbrs_scale = 0.3` but PBRS reward is nearly zero-mean

`reward/pbrs_approach` mean = −0.038. `reward/door_pbrs` mean = −0.008. These
are potential-based shaping terms (Ng-Harada-Russell). They're supposed to
guide the agent toward ideal positions without changing the optimal policy.

Empirically they contribute basically nothing (< 1% of reward magnitude).
Either the shape function is not producing the intended signal, or the scale
is too small. **Test**: log a histogram of these components per-step for one
episode. If both are always near zero, the shaping is inactive — probably a
bug in `_compute_pbrs` or the ideal-distance computation.

#### 2.2.4 (P2) `r_step: 0.0` disables the step penalty

Not a bug — an explicit design choice — but note that with `r_step = 0.0`,
the agent has NO reason to end episodes early. Only `clear_idle_extra` and
crash penalty push in that direction. If we fix #2.2.1 and remove
`crash_penalty` from expected-path terminations (#2.1.1), we may need a
small step penalty (say −0.001) to counteract the incentive to "stall"
for the entire 1800-step episode limit.

### §2.3 World Model (`isaac_models.py`, `vendor/networks.py`)

#### 2.3.1 (P1) KL is growing, not converging

Trajectory: `loss/kl` first=1.4, last=5.1, growth of 3.6 over 157 k steps.
This means the posterior and prior are diverging as training progresses.

Two hypotheses:

1. **Reward-driven regime shift**: as the agent's rollout distribution
   changes, the encoder needs to fit new obs statistics, and the prior lags
   behind. Once policy stabilizes (post-#2.1.1) the KL should re-converge.
2. **`kl_free_bits = 1.0` is too small**: paper default. But paper defaults
   assume a wider input space (Atari, DMC). Our input has many more
   dimensions (player, spatial, enemies, projectiles, pickups, doors,
   room_grid, passives — each with its own head loss). Free-bits per dim
   might need to be higher.

`kl_free_bits_frac` mean = 0.83 (0.86 last, 1.0 max, 0.016 min). So on
average, 83% of KL dims exceed the 1.0 nat floor. That means the free-bits
clip is **barely binding**. The KL loss is essentially unregularized.

**Fix (conditional on #2.1.1 being applied first)**: after we have stable
rewards, re-measure. If `kl_free_bits_frac` stays > 0.5 with the growing
KL trend, bump `kl_free_bits` to 2.0 or 3.0. If it drops below 0.2 (KL
gets pinned to the floor), keep the current value.

**Also**: `kl_dyn_scale = 0.5` and `kl_rep_scale = 0.1`. The dyn/rep split
is intentional (paper Fig 4). But `kl_dyn` and `kl_rep` in the logs are the
same value (they're both `kld(...).mean()` — see comment at
`vendor/networks.py:273-282`), so we can't verify the split is producing
different gradients in practice. **Add**: log the per-parameter-group grad
norms so we can see whether dyn_loss and rep_loss are actually driving
different sub-networks.

#### 2.3.2 (P2) Encoder loss dominated by a few heads

Reconstruction losses at end of run (`loss/{key}`):

- `passives`: 0.000026 (11k → essentially perfect)
- `player`: 0.19 (67 → good)
- `player_history`: 0.10 (1.25 → good)
- `room_grid`: 0.01 (239 → very good)
- `doors`: 0.016 (11.2 → good)
- `enemies_feats`: 0.054, `enemies_mask`: 0.065
- `projectiles_feats`: 0.008
- `pickups_feats`: 0.025

Recon losses drop by ~1000× from init. WM is fitting the data. The head
sizes may be unbalanced — `passives` and `room_grid` had initial losses of
111 and 239 respectively but only take a fraction of end-loss. The MLPs
per head may have very different capacity. Not a P0 issue, but worth
profiling: is the WM spending most of its update time on heads that are
already saturated?

### §2.4 Actor / Critic (`isaac_models.py:533-590`, `action.py`)

#### 2.4.1 (P1) Advantage magnitude is small and near-zero-mean

`loss/actor_adv_mean` = −0.001 (near zero, good — advantage should be
zero-centered by construction). `loss/actor_adv_std` = 0.066 (small).
`loss/actor_adv_abs_mean` = 0.032. Distribution suggests approximately
half-Normal with std ≈ 0.05 and a spike near zero.

Reinforce gradient magnitude: `|adv| * |dlog_prob/dtheta|`. With
`log_prob ≈ -3.8` (near uniform for `9x5` factors) and adv magnitude ~0.03,
the per-step gradient is order 0.1 — but this gradient is essentially
random-direction because adv sign is driven by imagination noise (see
#2.3.1), not by policy quality.

**This is why the actor doesn't consolidate.** It's not that the entropy
coefficient is wrong. It's not that `actor_lr` is too small. It's that the
gradient direction is noise-dominated. Fixing the reward signal (#2.1.1,
#2.2.1) will increase `adv_std` and give the reinforce term a real
direction to move.

#### 2.4.2 (P1) `actor_target_mean = -1.06` — imagined rewards are net negative

Imagined lambda-returns over `imag_horizon = 15` steps have a mean of −1.06
before EMA normalization. The critic has learned that from any state, the
expected return is negative — because most terminals are `crash_penalty = -1`.

This creates a subtle bias: **any action** the agent takes has an expected
outcome of about −1 within 15 steps. There's no way to be positive-net,
because the terminal state dominates the return. Even actions that avoid
death get pulled down by the eventual (imagined) crash.

Post-fix #2.1.1 (removing spurious crash penalties) this should improve
substantially. Also verify: `_compute_targets` uses `cfg.gamma = 0.997`,
which is fine for 15-step imagination (γ^15 = 0.956, barely decays).

#### 2.4.3 (P2) `MultiDiscreteDist.entropy` sums per-factor entropies

`action.py:120` — `entropy() = sum(d.entropy() for d in self._dists)`.
For factors [9, 5], max entropy = ln(9) + ln(5) = 3.81. Data shows
`actor_entropy = 3.807` initially → close to max, expected for uniform init.

**Concern**: the entropy is a SUM over factors, but the entropy BONUS in the
loss is `actor_entropy_coef * ent.mean()` — same coefficient regardless of
how many factors we have. If we later add a third factor (say 3-way "use
item"), the max entropy jumps to ~5, and the coefficient needs rescaling.
Not a current problem but a latent bug for future factor changes.

**Also**: the `entropy_bonus_mag` in metrics is `cfg.actor_entropy *
ent.mean() ≈ 3e-3 * 3.8 ≈ 0.011`. But this is the loss contribution, not
the gradient magnitude. The actual entropy gradient magnitude depends on
`dent/dtheta`, which is small when entropy is near-max (uniform policy).
So the entropy bonus effect on gradient is *even smaller* than the 0.011
number suggests. `reinforce_mag = 0.14` is 12× larger than `entropy_bonus_mag`,
but the gradient ratio is probably closer to 3-5× — still enough for
reinforce to dominate, but not the 12× the logs imply.

### §2.5 Config (`stage1_single_room_xs.yaml`)

#### 2.5.1 (P2) `batch_size = 8, seq_len = 32` → 256 samples per WM update

Small. Paper defaults are `16 × 64 = 1024`. Halving both is a 4× reduction in
gradient-signal-per-update. Combined with `train_ratio = 4` (4 WM updates per
env step), we get 1024 samples/env-step of gradient computation. Compared to
paper's `16 × 64 × 16 = 16k`, we're 16× lower.

This is intentional — XS variant for the 3060 Ti. But the effective sample
efficiency ends up being much worse than paper. Worth measuring whether
bumping `batch_size` to 16 (2× memory, ~1.4× time) is still within the
GPU budget after Track A's throughput improvements.

#### 2.5.2 (P2) `actor_entropy = 3e-3`

Paper default for continuous control: `3e-4`. Paper default for Atari
(discrete): also `3e-4`. Our value is 10× paper. Was raised on 2026-07-05
to prevent "entropy collapse". The observed data suggests we're on the
opposite end — entropy is stuck at max. May need to be tuned DOWN after
#2.1.1 and #2.2.1 are fixed and the actor starts actually consolidating.

Do **not** tune this now. Fix the reward-signal issues first, then measure
whether entropy is still stuck.

#### 2.5.3 (P1) `n_envs = 2` — episode correlation

With only 2 parallel environments, the replay buffer sees highly correlated
trajectory data. The WM's batched updates draw from replay at random
positions, but the underlying distribution is dominated by whatever the
most recent 2 rollouts looked like. **This is fine for on-policy methods
but unusual for a replay-based method like Dreamer.**

Paper uses `n_envs = 16` (Atari) or `n_envs = 4` (DMC). Isaac's compute
budget probably doesn't allow 16, but 4 would improve replay diversity
significantly (each of 4 rollouts is 50% shorter but produces 2× more
different states per unit time).

Rollback: if 4 envs causes GPU OOM or SPS drop below 2.0, revert to 2.

## §3. Assumptions I've been making that could be wrong

1. **The mod is deployed correctly.** — Verified once on 2026-07-08 (found
   at Steam path, contained `send_blocking`). But this needs re-verification
   after every mod-side change. Nothing in the build/deploy process makes
   this automatic.

2. **Isaac window focus doesn't matter after `send_blocking(2.0)`.** — Was
   the assumption when we bumped the mod's timeout. Later runs showed
   `mod_socket_error` still at 90%, and we're not sure why. The Isaac
   `log.txt` would answer this, but we've never seen it.

3. **The multi-discrete factored action space is being trained correctly.**
   — No end-to-end test proves the gradient flows through both factors.
   `MultiDiscreteDist.log_prob` sums per-factor log-probs, which is
   correct mathematically, but nothing verifies the gradient goes to
   both heads' parameters.

4. **`clear_idle_extra` is a helpful shaping signal.** — Was the
   assumption when set to −0.03. Actually acts like a policy-independent
   bias term (§2.2.1).

5. **`kl_free_bits = 1.0` matches the paper's regime.** — Paper defaults
   assume a stable reward signal and stable rollout distribution. Neither
   condition has held in our runs. Value may need to change.

6. **Imagination-based advantages have signal.** — Only true if the
   imagined trajectories are diverse enough that the critic can rank
   them. When most trajectories lead to `crash_penalty`, they're not
   distinguishable.

7. **Actor learning rate matters.** — Was going to recommend bumping.
   Actually, the LR is fine; what's wrong is the gradient DIRECTION
   (§2.4.1). No amount of LR tuning fixes noise-driven gradients.

8. **The 2026-07-06 baseline's "reward/death = -3 constant" meant deaths
   worked.** — Turns out the OLD logging only recorded terminal-tick
   values, which happened to be −3 whenever the mod's death event fired.
   `frac_nonzero` wasn't logged, so we couldn't see that death only fired
   in a fraction of episodes.

9. **`shaper_terminated_frac` accurately reflects the fraction of
   properly-terminated episodes.** — True after `c463321` added the
   metric. Not true before. Every claim about "old runs had death events"
   is based on the old (misleading) logging.

10. **HP-based death detection in `8d72114` works.** — Untested. This
    audit assumes it does; if it doesn't, add another entry.

## §4. Missing telemetry

Signals we should be logging but aren't:

### Actor / policy
1. **Per-factor entropy** — `factor_0_entropy`, `factor_1_entropy` (currently
   only sum is logged). If one factor collapses while the other stays
   uniform, we'd never know.
2. **Actor logit histogram** — distribution over which actions the policy
   currently favors. Right now we have no visibility into what the actor
   is DOING beyond "close to uniform".
3. **Per-action visit frequency** — how often each of the 45 (9×5)
   combined actions is chosen in the last N rollouts. Reveals mode
   collapse or dead actions.
4. **Advantage distribution shape** — currently we log mean/std/abs_mean
   but not skew, kurtosis, or a rough histogram. Would tell us if adv is
   Gaussian, bimodal, or spike-at-zero.

### World Model
5. **Per-dim KL** — we log the aggregate `kl_value.mean()`. If one dim is
   eating all the KL, `free_bits` per-dim needs adjustment. Log
   `(kl_value > free_bits).float().mean()` (done as `kl_free_bits_frac`),
   plus max and top-3 dim indices for the worst offenders.
6. **Reward-head calibration** — log a periodic `predicted_reward - actual`
   scatter over recent obs. If the WM is systematically under- or
   over-predicting, the imagination-based training is biased.
7. **Continue-head calibration** — same idea. `cont_head` predicts
   `1 - is_terminal`. Log its mean prediction on states where terminal
   actually happens.

### Env / mod
8. **Per-env crash counts** — right now `ep_end_reason` aggregates across
   `n_envs`. If one env is crashing more than the other (e.g., port 9500
   consistently but port 9501 not), that's a diagnostic clue.
9. **Time-to-first-obs after reset** — how long each `env.reset()` blocks
   waiting for the mod to reconnect. Anomalies show up as slow episodes.
10. **`send_blocking` return values** — the mod logs to Isaac DebugString
    but Python has no visibility. Add a Python-side counter incremented
    each time the mod's terminal-obs is `DELIVERED` vs `FAILED`. Requires
    a small mod protocol change (send a `delivered:true/false` in the
    terminal frame's metadata).

### Training loop
11. **Replay diversity** — sample entropy over stored obs. If the buffer
    fills with 500 k copies of the same 1000 states, we're overfitting to
    a narrow region.
12. **Grad norm per parameter group** — `wm_encoder`, `wm_dynamics`,
    `wm_heads`, `actor`, `critic`. Reveals if one component is grad-starved.

## §5. Prioritized rework plan

### P0 (must fix, blocks everything else)

**P0.1** — Split `crash_penalty` semantics. Do not apply `-1.0` when
`_try_accept_after_close` returned raw obs (mod cycled socket cleanly).
`env.py:275-311`.

**P0.2** — Fix `clear_idle_extra` dominance. Either gate on time-since-clear
or scale down by 5-10×. `reward.py:529`, `configs/*.yaml:reward.r_clear_room_idle_extra`.

Together these should shift `ep_reward` mean from −0.5 up to at least +2
(the current positive-side total), giving the actor a real gradient to
follow. `actor_target_mean = -1.06 → +1.5` roughly.

### P1 (do after P0, big impact)

**P1.1** — Verify `8d72114` (HP-based death detection) works. 15 min smoke
test. Success criteria: `shaper_terminated_frac > 0.5` and
`ep_end_mod_socket_error_frac < 0.2`.

**P1.2** — Add missing telemetry #1-#7 (per-factor entropy, actor logit
hist, adv distribution shape, per-dim KL, reward/cont calibration). We
cannot make further tuning decisions without these signals.

**P1.3** — Add hard timeout to `recv_exact` (protocol.py). Prevents hung
mods from hanging Python.

**P1.4** — Consider bumping `n_envs` to 4. Test SPS + GPU memory. If ok,
adopt.

**P1.5** — After P0.1 + P0.2 land, re-measure `kl_free_bits_frac` over
30 k steps. If still stuck at 0.7+, bump `kl_free_bits` to 2.0. If drops
below 0.2, leave at 1.0.

### P2 (polish, do last)

**P2.1** — Rework env.py's `_handle_crash_and_reaccept` to return
structured info (was-alive vs respawned) instead of just raw obs.

**P2.2** — Add end-to-end integration test: mock Isaac, simulate death
via HP=0, verify shaper terminates and trainer sees `shaper_terminated`.
Would have caught the "mod delivery" issue in July.

**P2.3** — Bump `batch_size` to 16 if GPU memory allows.

**P2.4** — Add step penalty `r_step = -0.001` after P0.1 fixes the terminal
signal. Prevents "stall to max_steps" degenerate policy.

**P2.5** — Log grad norms per parameter group.

## §6. What NOT to change now

Things that look suspicious but shouldn't be touched until the P0/P1 items
land, because they need the fixed baseline first:

- `actor_entropy = 3e-3`. Don't tune until entropy is actually moving.
- `actor_lr = 3e-5`. Not the problem. Direction, not magnitude.
- `imag_horizon = 15`. Fine.
- `gamma = 0.997`. Fine.
- Model sizes (encoder/decoder/RSSM). Fine.
- `train_ratio = 4`. Fine.
- `slow_target_fraction = 0.02`. Fine.
- `reward_ema = true`. Fine.

Touching these before the reward signal is real just adds a confound.

## §7. Estimated impact

If P0.1 + P0.2 both land clean, the qualitative training curve should shift
from "actor entropy 3.807 → 3.777 over 157k steps" to something like
"3.807 → 3.5-3.6 over 100k steps" — real consolidation. `ep_reward` should
go from `−0.5 → +2 to +5` in mean.

If the actor entropy STILL doesn't move meaningfully after P0.1 + P0.2, the
problem is deeper (WM prior is fundamentally broken, or there's a bug in
the reinforce math I haven't spotted). Only at that point should we start
tweaking model dimensions, KL weights, or LR.

## §8. One-page changeset preview

To make P0 concrete, here's approximately what the diff looks like:

```diff

+        # Terminate the episode with 0 penalty; HP-based death detection
```
