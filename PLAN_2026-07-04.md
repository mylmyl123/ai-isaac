# Path Forward — Focus on RL, Not Heuristic

**Date**: 2026-07-04. Written after 5 iterations chasing heuristic bugs.

## The realization

We spent 5 debug traces + 8 commits fixing heuristic bugs. Each fix
revealed another edge case. This isn't converging because **the
heuristic doesn't need to be good** — it just needs to give BC
"not-random" data. Isaac is stochastic and complex; hard-coding good
behavior is a dead end.

## What's now in place (v3, commit after this doc)

### Heuristic (drastically simplified)

3 rules, ~60 lines of logic. `python/isaac_rl/heuristic.py`.

  1. Enemy visible → shoot at nearest, sidestep perpendicular
  2. No enemy → move toward a random open door (locked per room)
  3. Room change → unlock target, re-pick

**Deleted**: stuck-detection, wall-rotation, failed-slot memory,
entry-slot inference, kite zones, projectile dodging, wall avoidance.

**Trade-off**: bot WILL sometimes get stuck against walls, ping-pong
between rooms, or die in dumb ways. BC learns these as valid states,
PPO's entropy exploration escapes them.

### RL signal (already tuned)

  - Reward shaping (room clear, new room, HP, PBRS, seek-door)
  - Idle-death penalty (-20 on prolonged idle)
  - Adaptive entropy floor (prevents policy collapse)
  - Weaker kickstart (0.2, 300-update decay) — BC influence decays fast

Export TB after 1M steps:
```powershell

```

**Success indicators**:
  - `rollout/ep_reward` trending UP (from strongly negative toward 0)
  - `rollout/ep_reward_best` climbing (record new highs)
  - `loss/entropy` staying above 1.0 (exploration alive)
  - `reward/new_room` firing regularly
  - `reward/room_clear` firing occasionally

**Warning indicators**:
  - `rollout/ep_reward` flat or trending down after 2M steps
  - `loss/entropy` collapsing below 0.5
  - `reward/idle_death` firing constantly (>50% of episodes)

### Priority 3: DAgger — human-in-the-loop demos (NOT YET WIRED)

`--human-override` records corrections to
`runs/<name>/human_corrections.npz`. But **these aren't yet used to
retrain BC**. TODO:

  1. Add `--merge-human-demos` flag to train.py
  2. Load human_corrections.npz alongside heuristic demos
  3. Weight human samples higher (they're higher quality)
  4. Retrain BC on merged data

This is where model quality comes from. The heuristic gives BC a start;
human corrections make it good.

**File to modify**: `python/isaac_rl/bc.py` — add a `merge_demos()`
function that concatenates heuristic + human data with configurable
weighting.

### Priority 4: If RL doesn't converge, diagnose at the RIGHT level

Do NOT patch the heuristic. Instead check:

  1. **Rewards**: does `rollout/ep_reward` show the shaping firing?
     Look at reward breakdown in the checkpoint (`reward_breakdown_last`).
     If room_clear rarely fires, maybe rooms are too hard for random policy.
  2. **Network**: is the encoder learning? Check `loss/value` — if it
     stays high (>2) after 1M steps, encoder isn't learning state.
  3. **Exploration**: is `loss/entropy` collapsing? Adaptive floor should
     prevent this. If it's collapsing anyway, raise `entropy_floor` to 1.2.
  4. **BC quality**: is BC teaching junk? Check `loss/kickstart` — if it
     grows monotonically fast, PPO is diverging from BC (which may be
     good if BC is bad).

## What NOT to do

  - ❌ Do NOT add more heuristic rules
  - ❌ Do NOT patch heuristic bugs revealed in debug traces
  - ❌ Do NOT expect the heuristic to be "good"
  - ❌ Do NOT collect more demos before verifying BC/PPO pipeline works

## Rationale

Every hour on the heuristic is an hour not on:
  - Reward shaping tuning (the actual RL signal)
  - DAgger workflow (human corrections)
  - Network architecture (encoder capacity, network depth)
  - Curriculum (stage1 → stage2 progression)

These are where 10x improvements come from. Heuristic gives at best 2x.
